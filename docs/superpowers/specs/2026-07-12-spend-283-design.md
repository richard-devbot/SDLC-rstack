# Spend policy, consumption, and provenance (#283)

## Decision

Spend is organized as a server-owned model with three deliberately separate concepts: current project policy, observed run consumption, and historical run policy snapshots. A configured limit is never counted as spend, zero-valued measured telemetry is never treated as missing, and a percentage is emitted only when both a real observation and a compatible cap exist.

## Enforcement truth

The run cap is enforced by `evaluateLoopBudget`: before another iteration, the harness stops when persisted `cumulative_cost_usd >= run_budget_usd`. There is no configured 80% warning contract, so the dashboard must not invent a “near” state. It reports `within_cap` until the exact `exhausted` boundary. Daily and monthly values are displayed as configured policy values, but are not described as loop-enforced until the backend publishes corresponding enforcement.

## Observation and provenance

A persisted `cumulative_tokens` object is the authoritative marker for persisted telemetry. A `metrics_write_failed` event makes persisted values stale and switches provenance to events, matching `resolveRunTotals` and client-state. A real zero is available telemetry. No marker and no `cost_recorded` event is unavailable.

Because the loop brake itself reads `metrics.json`, an event-derived observation after write drift is not a compatible numerator for cap position. The dashboard shows the actual event-derived consumption but marks enforcement position stale/unavailable and renders no percentage or remaining amount.

Every run observation carries cost, tokens, provenance, measurement time, project/run scope, current-cap position, and source. Project aggregates are computed only from runs in that project; global totals are the sum of those same observations.

Persisted measurement time is the `metrics.json` modification time and survives the rollup index. Profile and budget snapshots are compared independently so a missing profile file cannot suppress a valid historical-versus-current budget comparison.

## UX hierarchy

1. Current policy per project, with source, validity, freshness, and honest enforcement labels.
2. Actual observed consumption and provenance.
3. Per-run cap position using the exact loop boundary.
4. Historical policy snapshot versus current policy.
5. Stage and task drivers with textual equivalents.
6. Business Flex profile/routing detail as a secondary route.

At mobile widths, policy, actual spend, and remaining amount precede drivers. No chart relies on color alone.
